# Feature Research

**Domain:** Local-first analytics dashboard — SQLite persistence, background JSONL ingest, activity charts, cross-machine DB merge
**Researched:** 2026-03-19
**Confidence:** HIGH (architecture is well-understood; stack is locked; all features are internal engineering concerns with clear implementation paths, not user-facing unknowns)

---

## Context: What Already Exists

This is a subsequent milestone on an existing codebase. The following are **already built** and are NOT in scope for this research:

- Dashboard stats, daily/model charts, heatmap, peak hours
- Session list with search + pagination, session detail with conversation replay
- Project grid with per-project stats
- Cost analytics by model
- Active sessions with real-time JSONL tail-reading (5s poll — stays JSONL forever, not affected by this milestone)
- Data import/export ZIP, live/imported data source toggle
- JSONL parser (`reader.ts`), token/cost calculation, SWR hooks

The new milestone adds: SQLite persistence layer, background delta ingest, project activity chart, and DB file export/import/merge.

## Feature Landscape

### Table Stakes (Users Expect These)

Features required for the milestone to deliver on its stated goal. Missing any = the milestone is incomplete.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| SQLite schema matching existing types | All pages must query DB instead of re-parsing JSONL on every request — that is the entire point of the milestone | MEDIUM | Schema must cover `sessions`, `projects`, `daily_activity`, `model_usage`, `file_sync_log` tables. Columns derived directly from `SessionInfo`, `ProjectInfo`, `DashboardStats` types already in `types.ts`. No new data shapes needed. |
| Delta ingest: JSONL to SQLite (new/modified files only) | Full JSONL re-parse on every page load is the current performance problem being solved. Must not reparse unchanged files. | MEDIUM | Store `mtime` per file in a `file_sync_log` table. Compare filesystem `mtime` vs stored value. Only reparse files where `mtime` differs. Reuse existing `forEachJsonlLine` + `parseSessionFile` logic from `reader.ts`. |
| Background ingest trigger (periodic, automatic) | Ingest must happen without a user-initiated request. Users should not need to manually trigger a sync. | MEDIUM | Next.js App Router has no built-in background job runner. Use module-level `setInterval` inside a lazily-initialized singleton module. Runs once in the Node.js process on first API route import. Fire on startup and every 2 minutes. Known Next.js App Router issue: inconsistent singleton behavior in dev with hot reload — mitigate with `global` object cache. |
| All read API routes query SQLite | If API routes still scan JSONL, the milestone delivers no performance value. The data layer switch must be complete. | HIGH | Replace `getDashboardStats()`, `getProjects()`, `getSessions()`, `searchSessions()`, `getProjectSessions()` with DB-backed equivalents. Active sessions (`/api/active`) and session detail (`/api/sessions/[id]`) stay JSONL — they are excluded by design. |
| Session detail continues to read JSONL | Session messages (conversation replay) are too large and too ephemeral to store in SQLite — full message content is not needed in the DB. | LOW | `getSessionDetail()` in `reader.ts` continues to run as-is. Only session metadata rows go into the DB. No change required for the session detail page or its API route. |
| Export `.db` file download | User needs a way to transfer the analytics DB to another machine. | LOW | New `/api/export-db` GET route. Serve the `.db` binary file as a download. Complement (not replace) the existing ZIP export. The ZIP captures raw JSONL + settings + plans; the DB file captures parsed analytics only. |
| Import `.db` file upload | Counterpart to export — receiving machine must be able to load the file. | LOW | New `/api/import-db` POST route. Accept multipart upload, write to import directory, set data source flag so API routes read from the imported DB. Mirror pattern from existing `/api/import` ZIP route. |
| DB merge: deduplication by session ID | Core cross-machine use case. A user works on two machines and wants a unified analytics view. Session IDs are UUIDs generated by Claude Code — unique per session, stable, immutable. | MEDIUM | `ATTACH DATABASE 'other.db' AS other; INSERT OR IGNORE INTO sessions SELECT * FROM other.sessions;` — repeat for each table. `INSERT OR IGNORE` is correct because session data is immutable once written. No CRDT or conflict resolution needed. WAL mode required on both DBs to prevent locking during attach. |

### Differentiators (Competitive Advantage)

Features that make this milestone exceed expectations or add visible user value beyond the internal engineering work.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Activity chart on project detail page | Currently the project detail page has no time-dimension view — only aggregate totals and a flat session list. A sessions-per-day or messages-per-day bar chart makes per-project trends visible at a glance. This is the only new user-visible feature in the milestone. | LOW | Data is already available: `daily_activity` rows filtered by `project_id`. Use existing `BarChart` Recharts component pattern from the overview dashboard. No new chart type or library needed. Requires `project_id` column on `daily_activity` table — must be included in schema design. |
| WAL mode enabled by default | Prevents "database is locked" errors when the background ingest job runs concurrently with an API read. The alternative is sequential access only, which will cause visible request failures. | LOW | One line at DB init: `PRAGMA journal_mode=WAL`. No schema impact. WAL mode reduces per-transaction overhead and allows one writer + multiple concurrent readers. Should be treated as a required reliability measure, not an optional differentiator. |
| Ingest status indicator | User can see "last synced 30 seconds ago" without guessing whether the DB reflects recent Claude Code activity. Addresses the trust gap when transitioning from direct JSONL to DB-backed reads. | LOW | Expose last ingest timestamp and file count via `/api/db-status`. Display in the data management page or sidebar footer. Store last-ingest timestamp in an in-memory variable in the singleton ingest module — no DB write needed. |
| Merge preview before apply | Show count of new sessions that would be added before committing the merge. Prevents accidental data corruption if user merges the wrong file. | MEDIUM | Run merge selects inside a read transaction, count new rows per table, return summary JSON. Apply the actual `INSERT OR IGNORE` only after user confirms. Requires a two-step API: `POST /api/merge-db/preview` then `POST /api/merge-db/apply`. |

### Anti-Features (Commonly Requested, Often Problematic)

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| Full message content stored in SQLite | "Enable full-text search across all session messages from the DB" | Message content is large — a single long session can be several megabytes of JSONL. Storing it in SQLite rows would bloat the DB by 10x or more. Session detail page is already fast reading JSONL directly because it reads one file. | Keep JSONL as the message store. DB holds metadata only. If full-text search becomes a real user request, add SQLite FTS5 as a separate future milestone with a dedicated `message_fts` virtual table. |
| Real-time file watcher (fs.watch or chokidar) | "Ingest instantly when Claude writes a new file — eliminate the 2-minute lag" | `fs.watch` is unreliable on WSL (the primary deployment environment) — it misses events on network-mounted paths and has known Linux kernel event gaps. `chokidar` adds a dependency with the same underlying constraints. The user already accepted 5-second polling for active sessions — 2-minute ingest lag is appropriate for historical data. | Polling via `setInterval` every 2 minutes. Simple, reliable on WSL, no additional dependencies. |
| ORM or migration framework (Prisma, Drizzle) | "Type-safe queries and automatic schema migrations" | Adds build-time compilation step, native module complexity (Prisma generates platform-specific binaries), ORM learning curve, and schema introspection overhead. The DB schema for this milestone is 5-7 stable tables that will not change frequently. | `better-sqlite3` with raw SQL and `CREATE TABLE IF NOT EXISTS`. Track schema version with `PRAGMA user_version`. Manual migration functions in a `migrations.ts` file. Sufficient for this scope. |
| Cloud sync or CRDT-based sync | "Automatic cross-machine synchronization without manual export/import" | Contradicts the explicit local-first, no-accounts constraint in `PROJECT.md`. Adds auth, network dependency, server infrastructure, and ongoing maintenance cost. | Manual DB export/import/merge. The user explicitly controls when to sync. This is the intended design per the Key Decisions table in PROJECT.md. |
| Separate ingest process (worker_threads or child_process) | "Keep CPU-intensive ingest off the main Node.js thread to avoid impacting API response times" | JSONL ingest is I/O-bound (readline line-by-line), not CPU-bound. The delta check means only changed files are reparsed — typically 0-5 files per 2-minute cycle. Moving to a worker thread adds IPC serialization overhead, a second module lifecycle to manage, and a harder debugging surface with no measurable benefit at this data scale. | Run ingest as an async function in the same Node.js process. The existing `forEachJsonlLine` already uses async readline which yields to the event loop between lines. If ingest latency becomes measurable, revisit worker threads in a future phase. |
| Replace ZIP export with DB export | "Consolidate to one export format" | The ZIP export captures raw JSONL files, `stats-cache.json`, `settings.json`, plans, todos, and export metadata. The DB export captures only parsed analytics rows. They have different purposes: ZIP is a full data backup; DB file is analytics portability. Removing ZIP would eliminate the ability to restore raw session data. | Keep both. Surface them distinctly in the UI: ZIP = "Full backup", DB = "Analytics only". |

## Feature Dependencies

```
[SQLite schema + init]
    └──required by──> [Delta ingest job]
                          └──required by──> [API routes read from DB]
                                                └──required by──> [Project activity chart]
                                                └──required by──> [DB status indicator]

[Delta ingest job]
    └──enhances──> [Ingest status indicator]

[DB export route]
    └──pairs with──> [DB import route]
                         └──required by──> [DB merge]
                                               └──enhances──> [Merge preview]

[WAL mode pragma]
    └──required by──> [DB merge] (prevents locking during ATTACH)
    └──required by──> [Background ingest] (concurrent reads during ingest)
```

### Dependency Notes

- **Schema must be designed first.** Every other feature in this milestone depends on the DB existing with the correct tables and columns. Phase 1 must be schema + init + WAL mode.
- **`project_id` on `daily_activity` is a schema-level decision** that enables the project activity chart. If this column is omitted from the initial schema, the chart cannot be built without a schema migration. Include it in Phase 1.
- **Delta ingest must be proven before API migration.** Switching API routes to read from DB before ingest is verified to produce correct data would break all pages. Ingest correctness validation (comparing DB output to current JSONL-based output) must precede the migration cutover.
- **Import required before merge.** Merge operates on two DB files. Import brings the second file into scope as a local file. Without import, merge has no source file to work from.
- **WAL mode is a prerequisite for merge reliability.** SQLite's `ATTACH DATABASE` with concurrent reads/writes requires WAL mode to avoid locking. Enable at schema init, not as a later patch.
- **Ingest status indicator and merge preview are independent enhancements.** Neither blocks any other feature. They can be added at the end of the milestone or deferred to polish.

## MVP Definition

### Launch With (v1.1 core)

Minimum to deliver the milestone goal: pages load from DB, data is current, cross-machine portability exists.

- [ ] SQLite schema: `sessions`, `projects`, `daily_activity` (with `project_id`), `model_usage`, `file_sync_log` tables — WAL mode enabled at init
- [ ] Delta ingest job: scan `~/.claude/projects/`, compare file `mtime` to `file_sync_log`, reparse only changed/new files, upsert into DB — fires on module load and every 2 minutes
- [ ] All read API routes query DB (except `/api/active` and `/api/sessions/[id]`)
- [ ] Activity chart on project detail page — the only new user-facing feature in the milestone
- [ ] Export `.db` file download route
- [ ] Import `.db` file upload route
- [ ] DB merge with session-ID deduplication (`INSERT OR IGNORE`)

### Add After Core Validated (v1.1 polish)

- [ ] Ingest status indicator — adds transparency once ingest is proven stable
- [ ] Merge preview (row count before applying) — adds safety once merge is working
- [ ] Validation pass: compare DB-backed API output to JSONL-backed output on same dataset — confirms correctness before final cutover

### Future Consideration (v2+)

- [ ] Full-text search across message content (SQLite FTS5) — useful but large scope; defer until DB layer is battle-tested
- [ ] Worker thread for ingest — only if ingest measurably degrades API latency on real-world data volumes
- [ ] Scheduled ingest interval as user setting — hard-code 2 minutes for now; revisit if users report stale data
